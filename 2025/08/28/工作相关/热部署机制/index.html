类加载器隔离

类加载器隔离就是让不同的类加载器加载的类，即使全限定名相同，也被 JVM 视为不同的、互不相关的类。这种隔离机制创造了一种“命名空间”的效果。一个类由哪个类加载器加载，和这个类本身的全限定名（PackageName + ClassName）共同决定了它在 JVM 中的唯一性。
这个常见的场景：

  • 你有一个古老的 Web 应用（App A），它依赖于 commons-lang-2.5.jar。
  • 你开发了一个新的 Web 应用（App B），它依赖于 commons-lang-3.0.jar。
  • 这两个版本的 jar 包中都有 org.apache.commons.lang.StringUtils 这个类，但它们的内部实现和 API 可能完全不同。

类加载器的双亲委托模型

工作流程：

  1. 当一个类加载器收到类加载请求时，它首先不会自己去尝试加载。
  2. 它会将这个请求委托给父类加载器去完成。
  3. 父类加载器也会做同样的事情，继续向上委托，直到顶层的启动类加载器（Bootstrap ClassLoader）。
  4. 只有当所有父类加载器都反馈自己无法完成这个加载请求（它们的搜索范围中没有找到所需的类）时，子类加载器才会尝试自己加载。
    优势：
  • 安全性和稳定性：保证了核心 Java 类库（如 java.lang.String）总是由最顶层的启动类加载器加载，避免了用户自定义一个恶意的 java.lang.String 类来破坏核心库。
  • 避免重复加载：同一个类最多被加载一次（由父加载器加载），避免了重复加载。

实现隔离

常见的做法是重写 ClassLoader 的 loadClass(…) 方法（双亲委托的逻辑就在这个方法里），而不是只重写 findClass(…) 方法。
实现隔离的思路：

  1. 优先自己加载：对于特定范围（即需要隔离的 jar 包）内的类，在收到加载请求时，首先自己尝试加载，而不是先委托给父加载器。
  2. 无法加载再委托：如果自己找不到目标类，再 fallback 到双亲委托模型，让父加载器去尝试。
    以 Tomcat 为例：
  3. Tomcat 为每个部署的 Web 应用都创建了一个独立的 WebappClassLoader。
  4. 当 Web 应用中的代码需要加载一个类时（例如 org.apache.commons.lang.StringUtils），Tomcat 的 WebappClassLoader 会首先在自己的 Web 应用目录下的 /WEB-INF/lib 和 /WEB-INF/classes 中查找并加载。
  5. 如果找不到，它才会委托给父类加载器（Shared ClassLoader 和 Common ClassLoader等）去加载。
    这样，每个 Web 应用都有自己的类库副本，实现了应用级别的隔离。

隔离带来的挑战：类通信

隔离后，一个直接的问题是：不同类加载器加载的类如何交互？
例如，App A 中的 ControllerA 类和 App B 中的 ServiceB 类，由于被不同的 WebappClassLoader 加载，它们之间无法直接通过 import 和 new 的方式进行调用，因为 JVM 认为它们是两个不同的类，即使全限定名相同。
解决方案是使用接口和共享类加载器：

  1. 定义公共接口：将一个双方都认可的接口（例如 UserService）放在一个共享的目录（Tomcat 的 /lib 目录）中，由一个共同的父类加载器（如 Common ClassLoader） 加载。这个接口对所有 Web 应用都是可见的。
  2. 实现隔离：App A 和 App B 分别提供这个接口的实现类 UserServiceImplA 和 UserServiceImplB，它们被各自的 WebappClassLoader 加载，相互隔离。
  3. 通过接口交互：通过 Java 的反射机制、SPI（Service Provider Interface）或框架（如 Spring）的依赖注入，将实现类的实例绑定到接口上。因为接口是共享的，所以 App A 可以调用 (UserService) instanceA，即使 instanceA 的具体类是由另一个类加载器加载的，只要它实现了共同的接口即可。
    特性总结：
  • 核心思想 :不同的类加载器实例加载的相同全限定名的类，在 JVM 中被视为不同的类。
  • 主要目的 :解决类冲突，实现不同应用或组件使用不同版本库的共存。
  • 实现机制 :有控制地打破双亲委托模型，优先自己加载特定范围的类。
  • 典型应用 :Tomcat/Jetty 等 Web 容器（应用隔离），OSGi（模块化热部署），Spring Boot Executable Jar（嵌套JAR加载），Java Agent（ instrumentation ）。
  • 主要挑战 :被隔离的类之间不能直接通信，需要通过父加载器加载的共享接口和反射等机制间接通信。
